# Language

Follow project language conventions for documentation, comments, UI copy, PR descriptions, issues, and changelogs.

If a project has no convention, use English for user-facing and project artifacts.

For internal Yandex, Arcadia, Yandex Practicum, or other Yandex-related projects, use Russian by default for documentation, comments, commit messages, and project text unless the project says otherwise.

When writing in Russian, use Russian consistently. Translate explanatory prose, section headings, table cells, and bullet text. Do not mix English sentences or phrases into Russian documents.

Keep in English only:

- Proper nouns: product, company, project, and people names.
- Identifiers: code, API names, file paths, commands, config keys, package names, and protocol fields.
- Established domain terms without a stable Russian equivalent in the project or field (for example, harness, MCP, subagent) when the project convention or glossary says to keep them.

If the project defines a glossary or language style guide, follow it for borderline terms. Do not leave untranslated source phrases such as "Core insight" or "source of truth" unless they are quoted verbatim from a source in a blockquote.

Code, API identifiers, file names, commands, config keys, package names, and protocol fields stay in English.
